# 长视频剧情压缩剪辑系统计划

## 1. 目标

构建一个工具，把长视频、番剧或电影压缩成适合短视频平台分发的剧情解说/混剪视频。用户可以选择原始视频并设定目标时长，系统自动完成音频提取、台词转写、角色识别、剧情理解、压缩改写、AI 配音、时间戳回链和二次剪辑，最终输出一个故事完整、逻辑连贯的缩略版视频。

初期目标不追求完美复刻原角色音色，优先保证：

- 剧情主线完整。
- 人物关系和角色名稳定。
- 改写内容能追溯到原视频时间段。
- 生成的视频可直接用于人工复核和二次发布。

## 2. 核心使用流程

1. 用户导入一个或多个视频文件。
2. 用户设置目标输出：
   - 目标时长，例如 1 分钟、3 分钟、5 分钟、10 分钟。
   - 输出画幅，例如 16:9、9:16、1:1。
   - 解说风格，例如客观复述、短视频强节奏、悬疑推进、搞笑吐槽。
   - 是否保留原声关键台词。
   - 是否生成字幕。
3. 系统分析原视频：
   - 提取音频。
   - 识别台词文本。
   - 识别说话人。
   - 检测场景切换和镜头段落。
   - 建立剧情事件和时间戳索引。
4. 系统生成压缩剧情脚本。
5. 系统把压缩脚本映射回原视频片段。
6. 系统生成 AI 配音、字幕和剪辑决策表。
7. 系统调用 ffmpeg 生成成片。
8. 用户预览、修改脚本或片段后重新导出。

## 3. MVP 范围

第一阶段应做一个“可用但保守”的版本，避免一开始就投入过高难度的声音克隆和完全自动化影视级剪辑。

MVP 功能：

- 支持单个本地视频输入。
- 支持用户设定目标时长。
- 使用 ffmpeg / ffprobe 获取媒体信息并提取音频。
- 使用 ASR 模型生成带时间戳的转写文本。
- 使用说话人分离生成 speaker A/B/C 级别的角色轨迹。
- 使用 LLM 建立人物表、剧情事件表和故事大纲。
- 生成目标时长内的解说稿。
- 使用普通 AI TTS 生成旁白。
- 自动选择与解说稿对应的原视频片段。
- 输出横版或竖版视频。
- 输出可人工检查的中间文件，例如 transcript、角色表、剧情事件表、剪辑表。

MVP 暂不做：

- 原角色音色克隆。
- 逐字级口型匹配。
- 完全自动判断版权和平台合规。
- 多集连续剧全季级别记忆。
- 复杂转场、贴纸、花字和运营包装。

## 4. 推荐技术架构

建议采用“本地媒体处理 + AI 服务/本地模型 + 可审计中间数据”的架构。

```text
视频输入
  -> 媒体探测
  -> 音频提取
  -> ASR 转写
  -> 说话人分离
  -> 角色解析
  -> 场景/镜头检测
  -> 剧情事件抽取
  -> 目标时长压缩改写
  -> 时间戳回链
  -> TTS 配音
  -> 字幕生成
  -> ffmpeg 剪辑合成
  -> 成片输出
```

建议模块划分：

- `ingest`：视频导入、元数据读取、缓存管理。
- `audio`：音频提取、降噪、分轨、格式转换。
- `transcribe`：语音识别、词级/句级时间戳。
- `diarize`：说话人分离和 speaker 聚类。
- `story`：剧情事件抽取、人物关系、故事主线建模。
- `rewrite`：按目标时长生成解说稿。
- `align`：把解说稿和原视频时间段绑定。
- `voice`：TTS 配音。
- `edit`：剪辑决策表生成。
- `render`：ffmpeg 合成输出。
- `review`：人工校对和重生成入口。

## 6. 角色和台词一致性方案

这是项目成败的关键之一。建议不要只依赖一次性 LLM 总结，而是用多层证据建立“角色注册表”。

### 6.1 说话人分离

处理步骤：

1. ASR 先生成带时间戳文本。
2. diarization 模型把音频划分为 `spk_00`、`spk_01`、`spk_02` 等。
3. 将 ASR 句子和 speaker 时间段重叠匹配。
4. 对重叠、背景音、多人同时说话的片段标记为低置信度。

输出只应先得到 speaker，不要过早断言角色名。

### 6.2 角色名解析

角色名来源可以组合：

- 字幕或台词中直接出现的人名。
- 其他角色称呼。
- 片头片尾或文件名元信息。
- 用户可选手动标注。
- LLM 根据上下文推断。

每个 speaker 到 character 的绑定都必须保存证据和置信度。

### 6.3 人工校正入口

需要提供一个最小校正界面或配置文件：

```yaml
speakers:
  spk_00: 男主
  spk_01: 女主
  spk_02: 反派
```

原因：影视剧经常有背景音乐、哭喊、电话音、旁白、变声、群戏。全自动角色识别不稳定，MVP 必须允许人工修正。

### 6.4 改写约束

LLM 改写时必须输入结构化上下文：

- 人物表。
- 人物关系。
- 按时间顺序排列的 story beats。
- 关键原台词。
- 禁止改名规则。
- 目标时长对应的最大字数。

生成脚本时，每一句旁白都要带上来源 beat：

```json
{
  "line_id": "line_0018",
  "text": "女主终于发现，男主隐瞒的并不是感情，而是一场事故的真相。",
  "source_beat_ids": ["beat_0012"],
  "expected_duration": 5.2
}
```

这样后续才能稳定回链到原视频。

## 7. 时间戳回链和剪辑方案

不要让 AI 直接“猜”视频时间戳。正确做法是让 AI 只选择已经结构化的剧情事件，再由程序根据事件的原始时间范围生成剪辑表。

### 7.1 基础回链

每个 story beat 都保存：

- 原视频起止时间。
- 关联台词 id。
- 关联镜头范围。
- 重要度。
- 参与角色。

当脚本引用某个 beat 时，系统可直接得到候选片段。

### 7.2 片段选择原则

优先选择：

- 人物表情和动作明显的镜头。
- 关键冲突、反转、揭示真相的镜头。
- 与旁白含义强相关的画面。
- 开始和结束点不突兀的镜头。
- 时长足够支撑旁白，但不拖沓的镜头。

避免选择：

- 纯黑场、片头片尾、无意义转场。
- 长时间静止画面。
- 与旁白信息冲突的画面。
- 同一角色连续重复近景过多的片段。

### 7.3 时长控制

目标时长可以拆成：

```text
目标成片时长 = AI 旁白时长 + 原声保留片段时长 + 转场冗余
```

脚本生成前先估算字数：

- 中文普通 TTS：约每分钟 220 到 280 字。
- 情绪化短视频解说：约每分钟 260 到 330 字。

例如：

- 60 秒：约 230 到 300 字。
- 180 秒：约 750 到 900 字。
- 300 秒：约 1200 到 1500 字。

系统应先生成脚本，再用 TTS 实际音频时长校准，必要时自动压缩或扩写。

### 7.4 剪辑合成

ffmpeg 负责：

- 按 edit decision list 截取原视频。
- 拼接片段。
- 处理横转竖画幅裁剪。
- 混入 TTS 旁白。
- 对原始音频做 ducking 降低音量。
- 烧录或外挂字幕。
- 输出最终 mp4。

中间必须输出 EDL/JSON，便于人工检查。

## 8. 剧情完整性保障

为了避免“只剩爽点但故事断裂”，剧情压缩应分层处理。

### 8.1 故事层级

先建立四层结构：

1. 全片一句话主线。
2. 三幕或多幕结构。
3. 按时间顺序排列的关键剧情事件。
4. 每个事件下的原始台词和镜头证据。

压缩时先保留主线，再按目标时长删减支线。

### 8.2 必保剧情类型

通常必须保留：

- 主角目标。
- 主要冲突。
- 关键人物关系。
- 动机变化。
- 反转或真相揭示。
- 高潮。
- 结局或阶段性结论。

可优先删减：

- 重复铺垫。
- 与主线弱相关的支线。
- 纯氛围镜头。
- 低信息量寒暄。
- 可由一句旁白概括的过渡。

### 8.3 自动质检

生成脚本后再让 AI 做一次检查，但检查输入必须包含结构化原始大纲，而不是只看成稿。

检查项：

- 人名是否前后一致。
- 事件顺序是否和原片一致。
- 是否遗漏主线因果。
- 是否出现原片没有的事实。
- 旁白是否有无法回链的内容。
- 目标时长是否超出。

检查失败时，应回到 rewrite 或 align 阶段重试。

## 9. 配音方案

### 9.1 MVP

使用单一旁白 TTS：

- 生成一条完整解说音轨。
- 关键原声台词可保留 1 到 3 秒。
- 原视频背景音整体降低，避免和旁白冲突。

这是最稳的初期方案，也最符合短视频剧情解说形态。

### 9.2 第二阶段

支持多音色：

- 旁白一个音色。
- 男角色、女角色、反派等使用不同通用音色。
- 原始关键台词可以用字幕而不是重配。

### 9.3 第三阶段

再考虑角色音色学习或声音克隆：

- 需要每个角色足够干净的语音样本。
- 需要解决背景音乐和混响问题。
- 需要明确授权、版权和肖像/声音权益。
- 技术上要增加 speaker embedding、语音分离、音色训练或 voice conversion。

建议在产品中把声音克隆作为高级功能，而不是 MVP 阻塞项。

## 10. 画幅和短视频适配

短视频平台通常需要竖屏版本，因此需要单独设计画面处理。

可选策略：

- 横版原片直接居中裁剪为 9:16。
- 使用人物检测/人脸检测自动跟随主体。
- 上下模糊背景，中间保留原画。
- 关键字幕区域避开人脸和重要画面。

MVP 可先做：

- 16:9 原画输出。
- 9:16 模糊背景加居中主体。

后续再做智能裁剪。

## 11. 推荐实现阶段

### 阶段 0：原型验证

目标：证明一条视频能跑通从输入到成片。

交付物：

- 命令行工具。
- 一个样例视频的完整中间数据。
- 一条成片 mp4。
- 一份 EDL JSON。

验收标准：

- 用户输入视频和目标时长后可以生成视频。
- 成片剧情大体连续。
- 每句旁白都能追溯到一个或多个原片时间段。

### 阶段 1：MVP

目标：变成可重复使用的本地工具。

交付物：

- 本地 Web UI 或桌面 UI。
- 视频导入和参数设置。
- 任务队列和进度显示。
- 人物表校正。
- 脚本预览和编辑。
- 片段预览和替换。
- 一键导出。

验收标准：

- 非开发用户可以操作。
- 常见 20 到 60 分钟视频可完成分析和导出。
- 用户能修改错误角色名和错误片段。

### 阶段 2：质量增强

目标：提高自动化质量和短视频效果。

增强项：

- 更稳定的角色识别。
- 多集角色记忆。
- 智能竖屏裁剪。
- 高光镜头评分。
- 多版本脚本生成。
- 自动平台标题、简介、标签生成。
- 批量处理。

### 阶段 3：高级配音

目标：接近原剧多角色配音体验。

增强项：

- 多角色 TTS。
- 音色库。
- 授权前提下的声音克隆。
- 关键台词重配。
- 情绪控制。

## 12. 技术选型建议

媒体处理：

- ffmpeg / ffprobe。

ASR：

- 支持词级或句级时间戳的语音识别方案。
- 优先选择能和 diarization 对齐的方案。

说话人分离：

- diarization 模型或云服务。
- 需要输出 speaker 时间段和置信度。

剧情理解和改写：

- LLM 负责结构化抽取、压缩改写和质检。
- 必须使用 JSON schema 或强约束输出，避免纯文本难解析。

TTS：

- MVP 使用稳定的普通 TTS。
- 后续再接入多音色或克隆能力。

后端：

- Python 适合媒体处理和 AI pipeline。
- FastAPI 适合做本地 Web API。
- 任务队列可先用本地 SQLite + worker，后续再升级。

前端：

- MVP 可以用简单 Web UI。
- 重点是任务进度、脚本编辑、角色校正、片段预览，不需要复杂营销页。

存储：

- 原始视频不改动。
- 每个任务建立独立目录。
- 中间 JSON 全部落盘。
- 大文件走本地缓存。

## 14. 风险和应对

### 14.1 角色识别错误

风险：同一角色被分成多个 speaker，或多个角色被合并。

应对：

- 保存置信度。
- 提供人工合并/改名。
- 使用人物出场上下文和互称证据辅助判断。

### 14.2 改写出现幻觉

风险：AI 添加原片没有的情节。

应对：

- 每句脚本必须绑定 source beat。
- 质检时检查每个事实是否有来源。
- 不允许无法回链的剧情进入最终脚本。

### 14.3 剪辑不连贯

风险：画面跳跃、情绪断裂、镜头和旁白不匹配。

应对：

- 基于 scene boundary 截取，不在句中硬切。
- 给每个片段增加前后缓冲。
- 保留关键原声或反应镜头。
- 输出 EDL 供人工调整。

### 14.4 时长不准

风险：脚本字数估算和 TTS 实际时长不一致。

应对：

- 先生成 TTS 再二次校准剪辑。
- 允许自动压缩脚本。
- 给目标时长设置误差范围，例如 +/- 5%。

### 14.5 版权和平台合规

风险：影视内容二创、配音、声音克隆和平台发布可能涉及版权、邻接权、肖像权或声音权益。

应对：

- 产品层面提供合规提示。
- 声音克隆仅在用户确认有授权时启用。
- 保留来源记录，便于用户审查。
- 不把合规判断伪装成技术能力。

## 15. 初始实现里程碑

建议按以下顺序实施：

1. 建立任务目录和媒体探测。
2. 接入 ffmpeg 音频提取。
3. 接入 ASR，生成 transcript JSON。
4. 接入 diarization，合并说话人信息。
5. 生成基础人物表，允许手动修正。
6. 做场景切分和 story beat 抽取。
7. 按目标时长生成解说脚本。
8. 将脚本行绑定到 story beat。
9. 生成 TTS 音频。
10. 生成 EDL。
11. 用 ffmpeg 合成第一个版本。
12. 增加脚本和片段人工复核。
13. 增加竖屏输出。
14. 增加质检和重试机制。

## 16. 需要你确认的问题

这些问题会影响后续真正开始实现时的技术路线：

1. 你希望第一版是命令行工具、本地 Web 页面，还是桌面应用？
可以先最简单的形式，要先制作一个demo，验证整个流程可行性。
2. 第一版主要处理中文普通话视频，还是需要中英日等多语言？
可以多语言
3. 你更看重“剧情解说旁白版”，还是“保留大量原剧台词的精剪版”？
保留大量原剧台词的精剪版是我的最终追求，但是当前版本，我希望先做剧情解说旁白版，先验证整个流程可行性。
4. 目标视频常见长度是多少，例如 20 分钟番剧、45 分钟电视剧、还是 2 小时电影？
两小时多的电影是最终追求，但是当前版本，我希望先做20分钟左右的番剧，先验证整个流程可行性。

## 17. 我的建议

第一版建议做成“本地 Web UI + 后端 Python pipeline”的形态，而不是直接做复杂桌面软件。

原因：

- ffmpeg、ASR、diarization 和 TTS 的后端生态更适合 Python。
- Web UI 更容易做脚本预览、角色修正和片段复核。
- 未来可以平滑扩展为多人协作或云端任务。
- 本地处理视频文件更适合大体积素材，也更容易控制版权和隐私风险。

产品策略上，建议先做“AI 旁白剧情解说 + 原片画面混剪”，不要一开始做全角色复刻配音。这个路线更容易成功，也更符合短视频推流内容的实际形态。
